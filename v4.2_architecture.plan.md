# CodeWiser v4.2 — Architecture Plan

Builds on v4.1 (phase engine + 21-section depth) with **state**, **confidence/contradiction enforcement**, **schema-first export**, and the **Universal Bundle** (`MASTER_PROMPT.md` + canonical siblings + `_META.json`) as the **always-generated, complete primary deliverable** for **any** tool or model. **Tier-2 adapters** (~10 minutes copy-paste convenience) **never increase spec quality** and are **never required**. Shadow Critic remains a separate engine concern.

---

## §1 Stateful session layer

**Source of truth:** Supabase `session_state` (or equivalent). Every LLM turn reads state, writes deltas.

**`meta_log` shape (excerpt):**

```json
{
  "meta_log": {
    "phases_activated": [],
    "contradiction_resolutions": []
  }
}
```

- **`contradiction_resolutions`** — append-only array. When a contradiction flag is cleared (user-confirmed resolution or Delegation `ai_default` with `reversible: true`), archive the full resolution record here. **Never delete** entries; they mirror `_META.json` audit for exports.

Full row field list: `session_id`, `user_id`, `completed_phases`, `blueprint_blocks`, `complexity_level`, `reference_app`, `locale_context`, `storage_strategy`, `meta_log`, `confidence_map`, `contradiction_flags`, `fatigue_signals`, `created_at`, `updated_at`. Implement in `_data/session_state.schema.json`.

**`storage_strategy`:** `local_only | cloud_relational | cloud_document | local_first_sync_later` — captured in **`infrastructure/the_constraints`** (`session_state` row + `blueprint_blocks.constraints`). Drives **`data/schema_capture`** outputs and Pre-Export **Gate 3** (`_engine/pre_export_validator.md`).

## §2 Phase loader (summary)

`(current_phase, complexity_level) →` inject **≤2,000 tokens** of module text only. Never load the full `phase_library/` in one prompt. Spec: `_engine/phase_loader.md`.

## §3 Question priority engine (summary)

Each `question_patterns.md` entry carries `priority_weight: { impact, dependency, uncertainty }` (1–5). Sort `impact × dependency × uncertainty`; top 4–5 per exchange. Spec: `_engine/question_priority_engine.md`.

## §4 Completion engine (summary)

`completion_matrix.json` defines required fields + confidence thresholds per phase (and optional **activation / skip-for-complexity** metadata where the gate must no-op for skipped profiles). `ready_to_synthesize` is false until all **activated** phases pass gates. Spec: `_engine/completion_gate.md`.

**Example — ship layer success metrics (Medium+ only, embedded vs standalone per `codewiser_v4.1_refinement_7b662aba.plan.md` §4):**

```json
"ship/success_metrics_and_horizon": {
  "activation": "medium_and_above_only",
  "skip_for": ["minimal", "simple"],
  "required_fields": ["north_star_metric", "month_1_target"],
  "thresholds": {
    "north_star_metric": 0.6,
    "month_1_target": 0.5
  }
}
```

When `complexity_level` ∈ `skip_for`, the completion gate **does not require** these fields (`ready_to_synthesize` ignores this block for those sessions).

## §5 Contradiction rules engine

**Artifacts:** `_engine/contradiction_rules.json`, `_engine/contradiction_checker.md`.

**Top-level config in `contradiction_rules.json`:**

```json
{
  "max_open_contradiction_flags": 2,
  "resolution_policies": ["user_choice", "ai_proposal_reversible", "defer_to_blueprint", "escalate_shadow_critic", "ask_clarification"]
}
```

Full declarative rules live in `_engine/contradiction_rules.json` (including **`cross_field_rules`**).

**`contradiction_checker` behavior (post-phase or on write):**

1. **Max open flags:** At any time, `contradiction_flags.length` MUST NOT exceed `max_open_contradiction_flags` (2). New detections coalesce or replace per policy if at cap.

2. **Two-tier halt rule**
   - **`open_flags == 1`:** Insert a **resolution exchange at the START of the next phase** only. **No** global halt; the current phase may complete.
   - **`open_flags == 2`:** **Full halt.** Present **both** contradictions in **one** exchange. **No** new discovery questions until **both** are resolved or Delegation fallback applies.

3. **Delegation Detector fallback** — If the user **dismisses or ignores** resolution prompts **twice** while `open_flags == 2`: the engine **proposes both resolutions** as `source: ai_default`, sets **`reversible: true`** on each, **clears** `contradiction_flags`, **continues**. Log every proposed resolution to `_META.json` / `meta_log`.

4. **Clear condition** — A flag clears only when: **(a)** the user **explicitly confirms** a resolution, or **(b)** Delegation applies **`ai_default`** with **`reversible: true`**.

5. **Archive rule** — Resolved items are **appended** to **`meta_log.contradiction_resolutions[]`** (and export `_META`). **Never delete** archived rows.

Mirror **`max_open_contradiction_flags: 2`** and the two-tier rule in **`MASTER_ENGINE.md`** Golden Rules where contradictions are discussed.

## §7 Schema-first output

**Phase:** `phase_library/data/schema_capture/`. **`IMPLEMENTATION_PLAN.md`** is blocked until schema artifacts for the session’s **`storage_strategy`** are confirmed.

Read **`session_state.storage_strategy`** (and mirror in **`blueprint_blocks.constraints.storage_strategy`**). Branch:

### IF `storage_strategy == "local_only"`

- **Do not** generate **`SCHEMA.sql`**.
- Generate **`DATA_MODEL.md` only** — plain-English entity descriptions, field types, relationships.
- Generate **`SCHEMA.json`** with: `{"storage_type": "local", "engine": "<platform-appropriate>"}`. **`engine`** suggestions: **web → IndexedDB**; **React Native → expo-sqlite or AsyncStorage**; **Electron → SQLite**; **PWA → IndexedDB + Service Worker** (map from `blueprint_blocks.platform_access`).
- **`TECH_STACK.md`:** include recommended local persistence library for the chosen platform.
- **Pre-export Gate 3:** skip SQL primary-key / DDL checks; validate **`DATA_MODEL.md`** defines all entities referenced by the blueprint.

### IF `storage_strategy == "cloud_relational"`

- Generate **`SCHEMA.sql`** (full DDL) + **`SCHEMA.json`** (existing relational behaviour — no structural change vs baseline).
- **Pre-export Gate 3:** existing SQL completeness / PK rules apply.

### IF `storage_strategy == "cloud_document"`

- **Do not** generate **`SCHEMA.sql`**.
- Generate **`DATA_MODEL.md`** — collection names + document shape descriptions.
- Generate **`SCHEMA.json`** with: `{"storage_type": "document"}`.
- **Pre-export Gate 3:** every collection documents an **`_id`** field; skip SQL PK rule.

### IF `storage_strategy == "local_first_sync_later"`

- Generate **`DATA_MODEL.md`** + **`SCHEMA.json`** with `"storage_type": "local_first"` (and local **`engine`** hint as under `local_only`).
- **`TECH_STACK.md`:** add note — *Cloud sync is deferred. When ready for v1.1, consider Supabase for relational / Firebase for document — pick based on the data model.*
- **Pre-export Gate 3:** same as **`local_only`** (no **`SCHEMA.sql`**; validate **`DATA_MODEL.md`** completeness; skip SQL PK check).

## §10 Pre-export validator

**Artifact:** `_engine/pre_export_validator.md`.

**Blocking gates:** (1) required sections present, (2) zero **open** contradictions (`contradiction_flags` empty post-resolution), (3) schema completeness per `storage_strategy`, **(4) feature tier dependency consistency**, (5) e.g. budget alignment / contradiction archive integrity per bundle spec, **(6)** IMPLEMENTATION_PLAN chunking compliance (below).

### Gate 3 — schema completeness (`storage_strategy`)

Align with **§7** (duplicate for validator implementers):

| `storage_strategy` | Gate 3 behaviour |
| --- | --- |
| **`local_only`** | No **`SCHEMA.sql`** required. Require **`DATA_MODEL.md`** covering all blueprint entities; **no** SQL PK/DDL enforcement. **`SCHEMA.json`** must declare `storage_type: "local"` and platform **`engine`**. |
| **`cloud_relational`** | Require **`SCHEMA.sql`** + relational **`SCHEMA.json`**; standard SQL completeness and PK checks. |
| **`cloud_document`** | No **`SCHEMA.sql`**. Require **`DATA_MODEL.md`** per collection/document; **`SCHEMA.json`** has `storage_type: "document"`. Verify each collection defines **`_id`**. Skip SQL PK rule. |
| **`local_first_sync_later`** | Same as **`local_only`** for MVP export ( **`DATA_MODEL.md`** + **`SCHEMA.json`** with `storage_type: "local_first"` ); no **`SCHEMA.sql`**. |

### Gate 4 — `feature_dependency_map[]`

Emitted from **`phase_library/ship/feature_tiering/output_template.md`** after tier assignment. **Capture:** After tiers are set, run **one** AI-generated dependency pass; user **confirms or corrects** only (no manual graph).

**Per-feature entry schema:**

```json
{
  "feature_id": "string",
  "feature_name": "string",
  "tier": "MVP | v1.1 | Future | Drop",
  "depends_on": ["feature_id_1"],
  "depended_on_by": ["feature_id_3"],
  "dependency_type": "hard | soft",
  "ai_flagged": true
}
```

**Gate 4 rule:**

```
FOR EACH feature IN MVP tier:
  FOR EACH dep IN feature.depends_on:
    IF dep.tier != "MVP":
      BLOCK export — return structured conflict (which feature, which dependency, offending tier)
```

**Build-order note:** Pre-Export Validator **Position 9** depends on **`feature_tiering` output template** emitting **`feature_dependency_map[]`** (v4.1 `phase-ship` extension).

### Gate 6 — Implementation plan chunking compliance

Parse **`IMPLEMENTATION_PLAN.md`** (exported bundle). **BLOCK export** if any check fails.

| Check | Rule |
| --- | --- |
| Tasks per phase | No phase has **more than 7** tasks (count numbered **Tasks:** items under each `## Phase …`). |
| Phase count | **Total phases ≤ 10** (`## Phase` or equivalent headings). |
| Acceptance criteria | **Every phase** has **≥ 1** acceptance criterion (`Acceptance criteria` / checkbox list). |
| Phase 1 identity | Phase **1** is **Project Setup + Authentication**, **or** the plan documents an explicit waiver (why auth/setup is out of scope for this product). |

**Fail message:**

> Implementation plan has {N} tasks in Phase {X} — exceeds 7 task limit. Split into two phases before export.

Full operational text: **`_engine/pre_export_validator.md`**.

## §11 Universal bundle & MASTER_PROMPT

**Primary deliverable:** The engine **always** emits a **complete** **Universal Bundle**: **`MASTER_PROMPT.md`** (tool-agnostic instructions) **plus** up to **11** canonical sibling artifacts (see **`_output/bundle_spec.md`**) **plus** **`_META.json`**. **No tool adapter** is required for any vibe-coding product or model; the bundle is **self-sufficient**.

**Tier-2 adapters** (§12) are **optional** convenience packaging only.

**Bundle layout:** See **`_output/bundle_spec.md`** for the **Universal Bundle First Principle**, canonical filename table, conditional **`DESIGN_BRIEF.md`**, and **`storage_strategy`** handling for **`DATA_MODEL.md`** vs **`SCHEMA.sql`**.

**Bundle size (summary):** **`MASTER_PROMPT.md`** + numbered siblings ( **`DESIGN_BRIEF.md`** optional per `platform_access`) + **`_META.json`**.

### DESIGN_BRIEF.md — conditional generation

**Rule (mirrors `_output/bundle_spec.md`):**

- **Generate** `DESIGN_BRIEF.md` when `session_state.blueprint_blocks.platform_access` **includes any** of: `web`, `iOS`, `Android`, `PWA`, `desktop_app`, `cross_platform` (normalize `platform_access` to a list per `bundle_spec` if stored as a scalar).
- **Skip** `DESIGN_BRIEF.md` entirely when `platform_access` is **`api_only`** OR **`backend_only`**: do **not** emit the file, do **not** list it in **`MASTER_PROMPT.md`**, and set **`skipped_files`: `["DESIGN_BRIEF.md"]`** in **`_META.json`**.

### DESIGN_BRIEF.md (output template)

Exactly **four** fields; source phase **`design/look_and_feel`**. Maximum **one page**. **No** design tokens, component patterns, or accessibility rules — those are **v1.1** via Vision Accelerator.

| Field | Specification |
| --- | --- |
| **color_mood** | Single choice: `Minimal/Light` \| `Bold/Vibrant` \| `Dark/Dramatic` \| `Professional/Corporate` \| `Playful/Friendly` |
| **reference_ui** | String: `The UI should feel like [App Name]` **or** `No reference — AI should suggest`. Optional text input with skip button; not required for completion. |
| **font_feel** | Single choice: `Modern/Clean` \| `Classic/Traditional` \| `Friendly/Rounded` \| `Technical/Code-like` |
| **density** | Single choice: `Spacious/Open` \| `Balanced` \| `Compact/Dense` |

**Naming:** Use **`DESIGN_BRIEF.md`** only — do **not** use `DESIGN_SYSTEM.md` in the bundle or templates.

### IMPLEMENTATION_PLAN.md — generation rules (chunked for any LLM)

**Template file:** **`_output/IMPLEMENTATION_PLAN.template.md`** (structure + placeholders). Canonical rules below are normative.

**CORE PRINCIPLE**

- Each phase must be executable as a **standalone prompt** by the **weakest common** vibe-coding LLM (**equivalent to GPT-4o or Claude Haiku** quality).
- **Never assume** the tool has memory of prior phases beyond what is **explicitly stated** in the plan and sibling bundle files.

**PHASE SIZE LIMITS (hard rules)**

- Maximum **7** tasks per phase.
- Maximum **2,500 words** per phase when fully described.
- Maximum **10** phases for any app regardless of complexity.
- If **total tasks** across all phases **> 60:** merge adjacent phases until under the limit.
- Each phase must be completable in **one vibe-coding session** (approximately **≤ 1 hour** of build).

**PHASE COUNT BY `complexity_level`**

| Level | Phases (typical caps) | Max tasks per phase |
| --- | --- | --- |
| **minimal** | **3**, max **4** tasks each | (see Mandatory sequence below — minimal uses subset) |
| **simple** | **4–5**, max **5** tasks each | |
| **medium** | **6–7**, max **6** tasks each | |
| **complex** | **8–10**, max **7** tasks each | |

**MAPPING GUIDE (minimal / simple)**

- **Minimal apps:** Phases **1, 2, 4**, and **N** (final) — **four** phases total (setup+auth, data model, primary UI only, deployment checklist); merge or trim non-applicable mandated titles.
- **Simple apps:** Phases **1, 2, 3, 4**, **N** — **five** phases; Phase **3** may merge into Phase **2** if backend surface is trivial.
- **Medium apps:** Phases **1–6**, **N-1**, **N** (use mandatory names below; skip unused with explicit “Skipped — not applicable” only if emitter documents why).
- **Complex apps:** Phases **1–8**, **N-1**, **N**.

**MANDATORY PHASE SEQUENCE** (always this **order**; renumber sequentially in output)

1. **Phase 1:** Project Setup + Authentication  
2. **Phase 2:** Database / Data Model Setup  
3. **Phase 3:** Core API / Backend Logic  
4. **Phase 4:** Core UI — Primary Flow Only  
5. **Phase 5:** Secondary Features (**if Medium+**)  
6. **Phase 6:** Notifications + Integrations (**if applicable**)  
7. **Phase 7:** Admin / Control Panel (**if Complex**)  
8. **Phase 8:** Error Handling + Edge Cases  
9. **Phase N−1:** Testing + Bug Fixes  
10. **Phase N:** Deployment + Launch Checklist  

**MANDATORY PHASE STRUCTURE** — every phase must use this skeleton:

```markdown
## Phase N: [Name]
**Goal in one sentence:** [What this phase achieves]
**What you need from previous phases:**
[Explicit list of files/outputs that must exist before starting this phase. If Phase 1: "Nothing — start fresh."]
**Tasks:**
1. [Specific, atomic task — names files, functions, or components]
2. ...
**Files to create or modify in this phase:**
[Explicit list — no vague "create the backend"]
**Do NOT build in this phase:**
[Explicit deferrals — prevents scope creep]
**Acceptance criteria — check all before proceeding:**
- [ ] [Observable testable outcome]
- [ ]
**⛔ Stop here. Do not start Phase N+1 until all criteria above are checked.**
```

(max **7** tasks; max **5** acceptance bullets)

**REFERENCE ANCHORING (anti-hallucination)**

Include in **every phase** that touches the **data layer:**

> Refer to **`DATA_MODEL.md`** or **`SCHEMA.sql`** for all entity and field names. Do not create any table, collection, or field not defined there.

Include in **every phase** that touches **UI:**

> Refer to **`SCREEN_MAP.md`** for screen names and navigation flows. Refer to **`DESIGN_BRIEF.md`** for visual direction **(if present in bundle; omit DESIGN_BRIEF line if skipped per `bundle_spec`).**

Include in **every phase** that **adds product features:**

> Only build features tagged **MVP** in **`PRD.md`**. Stop and ask before building anything tagged **v1.1** or **Future**.

**PHASE HANDOFF NOTE** — append at end of **every phase except the last:**

> Before starting Phase N+1, confirm with the user: “Phase N is complete. The following work is done: [bullet list]. Ready to start Phase N+1: [name]? (Yes / No — say what needs fixing)”

See **§10 Gate 6** — export blocked if chunking rules are violated.

## §12 Tool adapter exports (Tier 2 — optional convenience only)

**Not required:** Every tool and model can consume the **canonical bundle** from §11 **as-is**. Adapters save ~**10 minutes** of copy-paste; they **do not** improve output quality.

**Adapter function pattern:** **`merge_and_format(bundle, target_tool)`** — **read** canonical bundle artifacts (`MASTER_PROMPT.md` + siblings + `_META.json`), **write** a tool-specific output package **only** under the user's chosen export folder. **Never** mutate or overwrite the canonical source bundle directory.

When the user opts in, adapters **omit** any merge that references a skipped file (**`DESIGN_BRIEF.md`** absent per `bundle_spec` ⇒ omit design-brief lines from stitched prose).

### Per-tool observed ingestion behavior

- **Cursor** — Ingests **`.cursorrules`** plus a **`/docs/`** folder of Markdown. **Optional adapter output:** rename **`MASTER_PROMPT.md`** → **`.cursorrules`**; copy all bundle **`.md`** siblings into **`/docs/`**. Format: long structured markdown with explicit file boundaries / references.

- **Lovable** — Single large prompt / knowledge-base upload. **Optional:** **`LOVABLE_CONTEXT.md`** stitching in dependency-aware order: **`PROJECT_BRIEF.md` → `PRD.md` → `USER_FLOWS.md` →** SCHEMA artifact (`DATA_MODEL.md` / `SCHEMA.sql` per bundle) **→ `IMPLEMENTATION_PLAN.md`** (omit skipped files). Format: flowing prose, clear section breaks, minimal nesting.

- **Bolt** — Split **system** vs **initial user** message. **Optional:** **`BOLT_SYSTEM.md`** = **`PROJECT_BRIEF.md` + `TECH_STACK.md` +** SCHEMA artifact, **maximum ~4,000 tokens** (truncate deterministic sections last if needed). **`BOLT_PROMPT.md`** = **`IMPLEMENTATION_PLAN.md` Phase 1 only** (first phase block). Format: concise, imperative, directive.

- **Rork** — Initial prompt plus **file attachments**. **Optional:** **`RORK_BRIEF.md`** = **`PROJECT_BRIEF.md` + `SCREEN_MAP.md` +** **`DESIGN_BRIEF.md`** when present (use bundle conditional rule; omit design section if skipped). **`SCHEMA.json`** attached separately. Format: screen-by-screen, component-level where helpful.

- **Replit** — **`README.md` at repo root** plus supporting docs. **Optional:** **`README.md`** = **`PROJECT_BRIEF.md` + `TECH_STACK.md` + `IMPLEMENTATION_PLAN.md` Phase 1**; copy **all other** bundle `.md` files into **`/docs/`**. Format: practical, environment-aware tone.

### Monitoring (backlog)

Quarterly smoke test of **each** adapter against real CodeWiser bundle output when adapters are maintained; adjust templates based on observed tool UI / limits.

**Spec files:** `_adapters/adapter_interface.md`, `_adapters/cursor_adapter_spec.md`, `lovable_…`, `bolt_…`, `rork_…`, `replit_…` (implementations follow this section).

## §13 Shadow Critic

**Artifact:** `_engine/shadow_critic.md`.

**Model (example):** Claude Sonnet-class second pass. **Output: JSON only:**

```json
{
  "critic_flags": [
    {
      "type": "contradiction | low_confidence | missing_required_field | logical_inconsistency",
      "field": "field_name",
      "description": "one sentence",
      "suggested_resolution": "one sentence"
    }
  ],
  "approved": true
}
```

**Trigger when ANY:**

1. `phase_transition: true` (phase just finished, before next phase questions)
2. `blueprint_blocks` updated AND any **newly written** field has `confidence < 0.7`
3. `contradiction_flags.length > 0` **after** current synthesis step
4. `ready_to_synthesize === true` (final pre-Blueprint check)

**Do NOT trigger for:**

- Mid-phase question generation with **no** `blueprint_blocks` write
- Simple confirmation exchanges (“yes”, “correct”)
- **Delegation Detector bulk-confirm** — **one** Critic call **at end of bulk**, not per field

**Behavior:** If `approved: false`, prepend **internal self-correction** to the engine’s next user-facing message; Critic output is **never** shown raw to the user. If `approved: true` and `critic_flags` empty, proceed normally.

**Cost estimate:** Simple (~8 phases) → **~5–7** Critic calls; Complex (~18 phases) → **~12–15** calls; roughly **+15–20%** session API cost (acceptable).
